# Slice based workflow

The goal of the repo is to create an AI worflow for automating the development of large scale taks or refactors.

The underlying idea is:

## RFC draft

One agent is given a task and it ask as many questions as needed to get clarity on the requirement/product/use case/refactor/etc. Based on this a first draft is created in markdown. Think of this as an RFC, PRD or other doc that outlines with enough detail what the user wants to build . It should include (but not limit to) motivation, goal to achieve, constraints, alternatives considered, potential impact, blast radius, external dependencies, architecture, etc. At this point the agent should ask the user if the work should happen on existing branch or if it should create a new branch.

## Draft polish

The draft is then passed to another agent with a fresh context window that refines the draft and returns a polished version, going deeper on impact, blas radius, alternatives and constraints comparing vs the codebase. Differs from the first agent in that this one runs autonomously without asking questions, doing specific updates (not full rewrites) on the draft. Similar to how a rfc review process goes.

## Plan

The polished doc is passed to another agent with a fresh context window that goes into plan mode and actually plans the required work.

This differs from the first agent in that it is not too concerned with requirement gathering or writing the RFC as with actually planning the work, meaning creating the actual tasks to acomplish the required work. The plan should be structured in what I call "slice-based workflow". It works by using the plan as guide along with extra documents for each "slice", it maps dependencies between slices and those that can be tackled in parallel. Take as sample a completed task [decouple-data-layer.md](implementations/decouple-data-layer/decouple-data-layer.md) along with the acompanying files. And another sample with an in progress task [income-tracking](implementations/income-tracking/income-tracking.md). Depending on the work, more files can be added but in general the work should be planned as follows:

1. The plan that outlines the high level goals, locked contracts and architecture.
2. A `progress` document that serves as both a checkpoint tracker and a **key decisions record**. After each slice, the agent appends key decisions (the WHY behind choices) to this document. Future agents read this to understand accumulated constraints without loading previous track files.
3. A `tracks` folder that contains specific information for each track. Pre-filled sections (scope, definition of done, validation, notes) are agent-facing. Post-execution sections (what was implemented, key decisions, open risks) are human-facing audit trail. Previous track files are **never loaded by future agents** -- they exist for human review only. Each agent loads only its own track file.
4. A `templates` folder that includes structured output that must be generated by every agent working on a specific slice to communicate with other agents. i.e the prompt for new agents or a reporting format.
5. Extra files can be created as long as they are helpful, lean and don't bloat the context window of new agents.

Context management rule: each slice agent loads exactly **3 files** plus explores code:
- Plan doc (goals, contracts)
- PROGRESS.md (key decisions record -- grows naturally, never capped)
- Current track file (scope, DoD, validation)
- Code exploration for understanding what exists (code is the source of truth for WHAT, documents carry the WHY)

Every new workflow should start with `slice 0` where the folder structure and the initial files are created, and end with `slice N` where a final check that everything has been achieved is done. That final slice should write a report file with key insights, relevant changes, and information that allow the user to understand WHAT was implemented, HOW it was implemented and WHY it was implemented that way.

Key considerations:

- Each slice's work should be small enough so that if fits inside a regular working session without triggering compaction (we are looking at filling at max 50% of the agent context window)
- Each slice's work should include its own validation: tests, quality checks, security, performance, etc
- If an agent considers that it should spawn subagents to perform tasks as code reviews, security reviews, performance reviews to prevent having its context window filled, it should do so.

The plan should be approved by the user before starting the next phase.

## Autonomous Slice Work

At this step agents with fresh context windows are spawned to work on the slices either in parallel or sequentially depending on the nature and dependencies of the slices. Each agents finalized work should trigger a new agent with fresh context window tackling the next slice. This part of the process should run autonomously without user intervention. This means each new agent should spawn with enough permissions to read files, edit files, do web search if needed and run commands. For security reasons and to prevent catastrophic operations the workflow should instruct the agents to focus the work ONLY on the current folder and repo the user is asking it to work. Also each agent should work on a different worktree and branch based off the initial working branch and when it is done it should merge its changes back.

i.e The original branch was feature/a, agent for slice 0 creates worktree feature/a-slice0 and merges back when it is done.

The prompt for those agents should be something like this:

```text
Objective:
Implement only `Slice 5 - Consolidation` for the decoupling effort.

Required context files (read first):
1) /Users/jcospina/personal/momo/implementations/decouple-data-layer/PROGRESS.md
2) /Users/jcospina/personal/momo/implementations/decouple-data-layer/decouple-data-layer.md
3) /Users/jcospina/personal/momo/implementations/decouple-data-layer/tracks/05-consolidation.md
4) /Users/jcospina/personal/momo/implementations/decouple-data-layer/migrated-ui-scopes.json

Execution rules:
- Follow the Simple Facade Approach plan exactly.
- Keep behavior unchanged (`payload + optional errorCode` compatibility).
- Respect server/client boundary safety and CSS safety rules.
- Implement only Slice 5; do not mix other slices unless strictly required.
- Reuse Slice 0 guardrails; do not remove them.

Implementation goals:
- Complete consolidation cleanup and strict boundary enforcement across migrated scopes.
- Remove or document remaining intentional direct transport imports in migrated areas.
- Ensure migrated UI layers use facades only.
- Keep any remaining direct transport usage intentionally scoped and documented.

Progress logging (mandatory):
- Update PROGRESS.md:
  - “What Is Implemented So Far”
  - “Current Focus”
  - Slice 5 checkbox status
- Update /Users/jcospina/personal/momo/implementations/decouple-data-layer/tracks/05-consolidation.md with:
  - what was implemented
  - key decisions
  - touched files
  - validations run + results
  - open risks/blockers
- Keep logs lean and actionable.

Validation rules:
- Run validations relevant to Slice 5 only.
- Run targeted tests for touched consolidation scope.
- Run `pnpm lint` and `pnpm format:check`.
- Do not run unnecessary broad commands.

Handoff requirement (mandatory):
- Summarize exactly what remains (if anything) after Slice 5.
```

But it can change depending on the nature of the work. Thats why it is key for the plan step to actually generate proper templates.

## Handoff

Once the last slice is done and the report is created, a PR should be created against the parent branch including the code along with the implementation notes. If there is no PR template in the repo then use the report as the PR description, otherwise summarize the report in the existing template and add a link to the file so the user can read it.

At this point the user can take two paths:

- Approve the PR and merge: no more work to do, the workflow comes to an end.
- Reject, leave comments or ask for changes: the workflow should react to those comments or rejection and spawn a new agent that takes in as context the changes on the PR including the implementation folder and the user's comments.
